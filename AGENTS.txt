# Developer Agent Guide (AGENTS.txt)

> **Audience: contributors.** This document is for agents **modifying** the
> `binarystruct` repository itself. If you are **using** the library in your own
> code, read **[llms-full.txt](llms-full.txt)** instead (indexed by `llms.txt`) —
> it has the API cheat sheet, recipes, and traps.

This document provides instructions for AI developer agents modifying, extending, or maintaining the `binarystruct` repository.

## 1. Ground Truth Specification
* **Crucial Rule**: The source of truth for all syntax constraints, interpreter mapping, and codegen compiler behaviors is **[SPECIFICATION.md](SPECIFICATION.md)**.
* When adding a new tag type, option, or keyword, follow the **Extension Protocol (Sync Checklist) in [SPECIFICATION.md](SPECIFICATION.md) §3** step by step. The critical invariant it encodes: the same behavior is implemented in **three parallel execution paths that must stay byte-identical**, and a change to one is a silent bug unless mirrored in all three:
  1. **Safe / reflection path** — `marshal.go` / `unmarshal.go` (parsed in `struct.go`).
  2. **Unsafe path** — `unsafe_io.go` (`unsafeWriteStruct` / `unsafeReadStruct`; active by default, replaced by panicking stubs under `-tags safe_binarystruct`).
  3. **Static codegen path** — `binarystruct-codegen/generator.go`.
* After implementing, add tests in **all three modes** (safe, unsafe, and the codegen integration suite) and update the docs: `SPECIFICATION.md`, `STRUCT_TAGS.md` (+ `STRUCT_TAGS_ja.md`), **`llms-full.txt`**, and the README recipe if it is a common pattern.
* **Performance numbers are generated, never hand-typed.** The cross-mode comparison table in the READMEs lives inside a `<!-- BENCH:START -->…<!-- BENCH:END -->` region produced by `make bench` (the `bench/` suite — safe vs unsafe vs codegen, with a `TestBenchParity` correctness guard). After a perf change, run `make bench` to refresh the region; do not edit it by hand. `make bench-smoke` just checks the benches still build/run in both modes (CI bitrot guard).
* **Deliberate codegen exclusions (do not "fix" as bugs).** A few features are intentionally runtime-only: the static generator emits a *clear generation error* and the struct falls back to the runtime interpreter. These are by design, not gaps to close — preserve the fail-loud error and runtime fallback rather than forcing byte-parity. Current exclusions: **multidimensional array tags over a non-scalar leaf** (`[2][3]string`, nested structs, pointers, or mixed fixed-array/slice nesting — codegen supports scalar-leaf multidim like `[2][3]int16`, but defers the rest to the runtime), struct-level `endian=inverse`, byte-order/encoding inheritance via embedding, a self-referential `valueof=bytelen(F)` cycle, and a **custom `valueof` evaluator over a nested-struct arg** (all other arg shapes are supported — byte regions and integer scalars are emitted inline; text-encoded/prefixed strings, floats, multibyte-scalar arrays, padded byte slices, and variable string buffers are re-encoded via `ms.MarshalAs`; only a nested struct fails generation). When adding a feature that codegen can't represent, follow this same pattern (fail loud + documented limitation) instead of generating incorrect code.

## 2. Codebase Architecture Map
* **[struct.go](struct.go)**: Layout parser and AST-like metadata compiler (`getStructMetadata`).
* **[marshal.go](marshal.go) / [unmarshal.go](unmarshal.go)**: Reflection-based safe-mode runtime interpreter.
* **[unsafe_io.go](unsafe_io.go)**: Reflection-free, high-performance unsafe pointer interpreter.
* **[simd_amd64.go](simd_amd64.go) / [simd_fallback.go](simd_fallback.go)**: Optional SIMD byte-swap kernels behind the `experiment_simd` build tag (`GOEXPERIMENT=simd -tags experiment_simd`). `unsafe_io.go`'s arch-neutral `swapBytes` dispatches to `simdSwap16/32/64`; the amd64 file implements them with an AVX2 `VPSHUFB` shuffle, the fallback with the scalar loop. `swap_test.go` validates the dispatcher against an independent reference and runs in **both** builds.
* **[binarystruct-codegen/](binarystruct-codegen/)**: Separate compiler module generating static methods to bypass interpreters.
* **[STRUCT_TAGS.md](STRUCT_TAGS.md)**: User-facing documentation for tag syntaxes.

> **Upstream-tracking note (SIMD).** The vectorized kernel is **amd64-only by design** because Go 1.26's `simd/archsimd` exposes vector types/ops for amd64 *only* (every functional file in the package is `_amd64.go`; referencing a vector symbol under `GOARCH=arm64` fails to compile). All other architectures intentionally stay on the scalar `simd_fallback.go`. **If a future Go release adds another architecture to `simd/archsimd` (arm64 NEON is the likely first), update this code to match:** add a `simd_<arch>.go` implementing `simdSwap16/32/64` with that ISA's primitives (e.g. NEON `REV16/REV32/REV64`, which byte-reverse within elements directly — not a copy of the amd64 shuffle), and narrow `simd_fallback.go`'s build-tag negation to exclude the new arch. The dispatcher (`swapBytes`) and `swap_test.go` are already architecture-agnostic, so no other file needs to change. Until then, keep SIMD documented and scoped as amd64-only.

## 3. Development and Testing Rules
* **Build, test, and verify** (a `go.work` two-module workspace — the root library and the nested `binarystruct-codegen` module). All of the following must pass before a change is complete:
  ```bash
  go test ./...                              # default = unsafe interpreter path + codegen integration
  go test -tags safe_binarystruct ./...      # safe / reflection-only path (must match)
  (cd binarystruct-codegen && go test ./...) # the code generator's own module
  gofmt -l . && go vet ./...                 # formatting + vet must be clean
  ```
* **Test Hygiene**: Keep the default test output quiet. Any subprocesses or integration tests (like [codegen_integration_test.go](codegen_integration_test.go)) must respect the `testing.Verbose()` flag and capture outputs instead of writing directly to stdout/stderr.
* **Compatibility**: Ensure any optimizer or codegen enhancements fall back gracefully to the safe reflection mode if compiled with `-tags safe_binarystruct`.
